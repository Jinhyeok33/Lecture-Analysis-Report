# LLMEngine v2 최종 수정 필요사항

## 문서 범위

- 이 문서는 `ik/LLMEngine`만 대상으로 한다.
- `ik/Lecture-Analysis-Report/LLMEngine`는 v2 sub(백업용)으로 간주하며, 본 문서의 수정 대상에서 제외한다.
- 아래 내용은 "기능 추가 아이디어"가 아니라, v2를 실무 배포 가능한 수준으로 끌어올리기 위해 반드시 정리하거나 수정해야 하는 항목만 모은 것이다.

## 요약

현재 v2는 구조 분리는 어느 정도 되어 있지만, 운영 관점에서는 아직 위험하다. 가장 큰 문제는 다음 네 가지다.

1. 실패를 실패로 남기지 않고 정상 결과처럼 섞어 넣는 구조
2. 체크포인트/재실행/비동기 처리 설계가 실제 운영 시 신뢰성을 보장하지 못하는 구조
3. 입력 포맷과 팀 간 인터페이스 계약이 문서화되어 있지 않아 upstream 변경에 매우 취약한 구조
4. LLM 평가 프롬프트, evidence 검증, 집계 결과 계약이 서로 완전히 일치하지 않는 구조
5. 생성형 AI를 결정론적 시스템처럼 가정하고 "100% 통과"를 목표로 삼는 비현실적 운영 관점
6. 비용, 토큰, 프롬프트 버전, 민감정보 처리에 대한 운영 메타데이터가 비어 있는 구조

즉, 현재 v2의 우선 과제는 "정확도 향상"보다 "실패 모델 정립", "입출력 계약 고정", "운영 복구 가능성 확보"다.

---

## 1. Critical Risks

### 1-1. 실패 은폐: fallback 결과가 정상 결과처럼 집계됨

현재 구조에서는 청크 분석 실패 시 `FAILED` 상태를 저장하더라도, 실제 출력 JSON에는 중립 점수(주로 3점대) fallback 결과가 정상 청크처럼 포함될 수 있다. 이 방식은 다음 문제를 만든다.

- OpenAI 인증 실패, Rate Limit, Timeout, evidence 검증 실패가 모두 "평범한 강의 품질"처럼 보일 수 있음
- 최종 summary가 실제 강의 품질이 아니라 장애 상황의 산물로 오염될 수 있음
- 운영자가 결과 JSON만 보고는 어떤 청크가 실패했는지 알 수 없음

### 1-2. 체크포인트/재실행 전략 불일치

현재 v2는 checkpoint를 저장하지만, 실행 시점 lecture_id/run 번호 정책이 일관되지 않으면 같은 강의를 재시작해도 기존 checkpoint를 제대로 재사용하지 못한다. 이 구조는 아래와 같은 운영 문제를 만든다.

- 중단 후 재실행 시 이전 진행 상황을 잇지 못하고 처음부터 다시 분석할 수 있음
- output 파일 이름과 checkpoint 파일 이름 사이의 관계가 불안정해질 수 있음
- run 번호 규칙이 서비스 레이어와 entrypoint 레이어에서 다르면 장애 복구가 어려워짐

### 1-3. 비동기 처리 병목 및 가짜 동시성

비동기 호출이 존재하더라도, 그 안에서 동기 파일 I/O와 전역 잠금이 섞여 있으면 실제 처리량은 쉽게 무너진다.

- 청크 상태 저장이 전체 JSON 파일 재읽기/재쓰기 기반이면 concurrency를 높일수록 병목이 심해짐
- event loop 내부에서 동기 저장소 호출이 계속 발생하면 "비동기"의 이점이 크게 줄어듦
- 로컬 JSON 저장소는 운영 확장성 면에서 테스트용 또는 임시용에 가깝고, 실서비스용 지속성 계층으로 보기 어려움

### 1-4. 체크포인트 파일 손상 시 조용한 데이터 유실 가능성

체크포인트 JSON이 조금만 깨져도 상태를 비우거나 무시하는 식의 처리라면, 시스템은 "장애를 감지"하지 않고 "기존 진행 상황을 잃고 조용히 재시작"하게 된다. 이는 특히 배치 처리에서 매우 위험하다.

- 부분 쓰기 실패
- 강제 종료
- 동시 접근
- 인코딩/파일 손상

위 상황에서 checkpoint 복구 전략이 없으면 운영 신뢰성이 급격히 떨어진다.

### 1-5. 입력 포맷 변경에 매우 취약한 chunk parser

`chunk_processor.py`는 특정 STT 포맷을 기대한다. 만약 팀원1이 전처리 과정에서 시간 정보나 화자 ID를 제거하거나 필드 순서를 바꾸면, v2는 시작부터 오동작할 가능성이 높다.

- `<HH:MM:SS> 화자ID: 발화` 형식이 조금만 바뀌어도 파싱 실패
- 파싱 실패 라인을 이전 발화에 무조건 이어붙이는 방식은 speaker/timestamp 오염을 유발
- STT 전처리 로직이 바뀌면 downstream evidence 검증까지 연쇄적으로 흔들림

### 1-6. 13개 평가 항목 동시 처리로 인한 컨텍스트 희석

한 번의 프롬프트에 13개 평가 기준을 모두 실으면 LLM attention이 분산되고, 아래 문제가 생길 수 있다.

- 일부 항목은 무성의한 3점 처리
- 근거 없는 strengths/issues 생성
- evidence 누락 또는 환각
- 항목 간 기준 충돌

특히 일부 항목을 NLP 모듈이 맡는 구조라면, LLM 결과와 NLP 결과를 어떤 기준으로 병합할지 v2 문서와 코드에서 명확히 정의해야 한다.

### 1-7. Evidence 원문 substring 검증 고정은 위험

설계 문서나 협업 문서에 "evidence는 원문 substring 검증"처럼 적혀 있다면 v2 기준으로 반드시 수정해야 한다. LLM은 원문을 거의 맞게 찾아도 조사, 띄어쓰기, 어미를 미세하게 바꾸는 경우가 흔하다.

- strict substring만 허용하면 정답에 가까운 evidence도 실패 처리될 수 있음
- 실패가 누적되면 재시도 루프가 길어지고 비용이 커짐
- 실제로는 맞는 응답인데 시스템이 틀렸다고 판단하는 false negative가 많아짐

참고로 v2 코드에는 fuzzy 성격의 방어가 일부 들어가 있더라도, 문서/설계/운영 정책이 여전히 strict substring 중심으로 남아 있으면 팀 전체 구현 방향이 다시 잘못 굳어질 수 있다.

### 1-8. 인터페이스 Spec-out 없이 병렬 개발하는 방식 자체가 통합 리스크

`hs_nlp`가 전처리한 결과를 v2가 받아 청킹하고, 그 결과를 다른 모듈이 다시 소비하는 구조라면, 코드 작성보다 먼저 입출력 계약을 확정해야 한다. 이 계약 없이 "각자 교류 없이 진행"하는 방식은 통합 테스트 시점에 결합 리스크가 한꺼번에 폭발하는 전형적인 실패 패턴이다.

- upstream가 `<HH:MM:SS>`를 `[HH:MM:SS]`로만 바꿔도 파서가 바로 흔들릴 수 있음
- 화자 ID 뒤 공백, 구분자, 줄바꿈 정책만 달라져도 청킹과 evidence 검증이 연쇄적으로 깨질 수 있음
- 계약 없는 상태에서는 버그의 원인이 코드인지 데이터인지 팀이 밤새도 분리하기 어려움

이 문제는 단순 커뮤니케이션 이슈가 아니라 데이터 파이프라인의 핵심 설계 결함으로 봐야 한다.

### 1-9. "통과율 100%"를 목표로 잡는 것은 잘못된 KPI

LLM은 결정론적 함수가 아니라 확률론적 생성기다. Structured Outputs, schema validation, prompt tuning을 적용하더라도 아래 상황은 항상 일정 비율로 발생한다.

- schema 파손
- refusal
- 빈 응답 또는 부분 누락
- hallucinated evidence
- 재시도 후에도 복구되지 않는 응답

따라서 v2는 "100% 성공"을 전제로 설계하면 안 된다. 올바른 목표는 "실패가 발생했을 때 파이프라인이 어디까지 유지되고, 어떤 상태 코드로 복구 가능하게 남는가"다.

- 실패 청크를 버릴지
- 기본값 점수를 넣을지
- `result_integrator` 또는 downstream에 어떤 상태를 넘길지
- 재시도 한도 이후 어떤 degraded 결과를 남길지

이 네 가지가 명확해야 한다.

### 1-10. 비용 및 토큰 사용량이 보이지 않는 것은 운영 리스크

현재 문서에는 에러 추적과 상태 추적은 들어가 있지만, 비용 관측성은 비어 있다. 그러나 LLM 파이프라인은 기능 문제만큼 비용 폭주가 치명적이다.

- 재시도 루프가 길어지면 하룻밤 사이 비용이 급증할 수 있음
- 강의 1건당 분석 비용을 알 수 없으면 최적화 우선순위를 정할 수 없음
- 프롬프트 변경 전후 비용 비교가 불가능함

따라서 v2는 최소한 강의 1건 단위로 아래 메타데이터를 남겨야 한다.

- total_prompt_tokens
- total_completion_tokens
- total_tokens
- estimated_cost
- llm_call_count

이 데이터가 없으면 운영자는 성능은커녕 예산 통제도 할 수 없다.

### 1-11. 개인정보 및 민감정보 마스킹 부재는 보안 리스크

STT 스크립트에는 강사나 학생이 무심코 말한 개인정보가 섞일 수 있다. 학생 이름, 전화번호, 이메일, 특정 기업 내부 정보가 원문 그대로 외부 LLM API로 전송되면 기능 문제가 아니라 보안/컴플라이언스 문제가 된다.

- 학생 이름
- 전화번호
- 이메일
- 사번/고객번호 등 식별자
- 기업 내부 시스템명, 고객명, 민감한 내부 발화

따라서 v2는 "분석 정확도" 이전에 "외부 전송 가능 데이터인가"를 먼저 판단해야 한다.

### 1-12. 프롬프트 형상 관리 부재는 품질 회귀 추적 불가로 이어짐

프롬프트는 코드와 동일한 수준의 버전 관리 대상이다. 그런데 분석 결과 메타데이터에 어떤 프롬프트 버전이 쓰였는지 남지 않으면, 결과 품질이 흔들렸을 때 원인을 추적하기 어렵다.

- 같은 코드라도 prompt 문구 한 줄 차이로 평가 품질이 크게 변할 수 있음
- 이상치 결과가 나와도 어떤 프롬프트 버전에서 생성됐는지 구분할 수 없음
- 이전 프롬프트로 롤백하거나 A/B 비교하기 어려움

따라서 최종 결과에는 최소한 `prompt_version`이 남아야 하며, 가능하면 `model_version`과 함께 묶어서 추적해야 한다.

## 2. 예상되는 오류 시나리오

### 2-1. Upstream 포맷 변경으로 인한 초기 파싱 붕괴

- 시간 정보 누락
- 화자 ID 누락
- 콜론 위치 변경
- 여러 줄 발화 포맷 변경

이 중 하나만 발생해도 청크 분할, evidence 검증, 집계 결과가 모두 흔들릴 수 있다.

### 2-2. LLM API 장애

- HTTP 429 Rate Limit
- HTTP 500/502/503 계열 서버 장애
- Timeout
- 네트워크 단절
- 인증 실패
- quota 초과

단순 `try-except`만으로는 부족하다. 실패 유형별 재시도 가능 여부를 구분해야 한다.

### 2-3. Content Filter / Refusal

LLM이 안전 정책이나 내부 refusal로 응답을 거부하면 구조화 출력 자체가 비어 버릴 수 있다. 이 경우를 별도 상태로 분리하지 않으면 일반 실패와 동일하게 취급되어 분석 품질을 왜곡할 수 있다.

### 2-4. 부분 누락 응답

LLM이 아래 중 일부를 누락할 수 있다.

- 특정 항목 점수
- evidence 배열
- strengths / issues
- 구조화 출력 일부 필드

이 상황에서 `aggregator.py`가 분모 0, 잘못된 평균, KeyError 없이 돌아가도록 방어되어야 한다.

### 2-5. 청크별 결과 길이/계약 불일치

프롬프트는 "정확히 10개"를 요구해도 실제 schema나 fallback은 8~15개를 허용하면 downstream 소비자는 안정적으로 붙을 수 없다.

## 3. 반드시 문서화해야 할 인터페이스 계약

코드보다 먼저 확정되어야 할 것은 팀 간 데이터 계약이다. 다만 현재 단계에서는 이미 사용 중인 `dataset/강의 스크립트/*.txt`가 사실상 입력 기준 역할을 하고 있으므로, 이를 임시 입력 표준(working contract)으로 간주하고 v2를 보강하는 접근이 현실적이다. 이후 팀원1(hs_nlp), 팀원3와 연동 시 아래 계약을 정식 문서로 확정하면 된다.

### 3-1. 팀원1 -> v2 입력 계약

현재 단계에서는 `dataset/강의 스크립트/*.txt`의 실제 형식을 임시 표준으로 삼고, 이후 필요 시 아래 중 하나를 정식 계약으로 고정한다.

- TXT 표준 포맷
- JSON 표준 포맷

최소 필드는 아래 수준으로 고정하는 것이 안전하다.

```json
{
  "lecture_id": "string",
  "source_file": "string",
  "lines": [
    {
      "timestamp": "HH:MM:SS",
      "speaker_id": "string",
      "text": "string"
    }
  ]
}
```

필수 합의 항목:

- timestamp 필수 여부
- speaker_id 필수 여부
- 허용 인코딩
- 줄바꿈 정책
- 빈 줄 처리 방식
- STT 노이즈 제거 규칙
- 중복 제거 규칙
- 한 줄이 여러 문장일 때 처리 기준
- 개인정보(PII) 마스킹 완료 여부
- 외부 LLM 전송 가능 수준으로 비식별화했는지 여부

### 3-2. v2 -> 팀원3 출력 계약

`_chunks.json`, `_summary.json`, checkpoint, 최종 출력 파일 각각에 대해 다음을 고정해야 한다.

- 파일명 규칙
- lecture_id 규칙
- run_id 규칙
- 필수 필드
- 선택 필드
- null 허용 정책
- 배열 길이 보장

### 3-3. v2 내부 모듈 계약

v2 내부에서도 모듈 간 계약이 고정되어야 한다.

- `chunk_processor.py`는 어떤 `ChunkMetadata`를 보장하는가
- `openai_adapter.py`는 실패 시 어떤 예외/상태를 반환하는가
- `aggregator.py`는 누락 점수와 실패 청크를 어떻게 다루는가
- `json_repo.py`는 어떤 상태 전이를 보장하는가

### 3-4. Schema Contract 운영 규칙

명세서는 한 번 쓰고 끝나는 문서가 아니라 변경 통제 대상이어야 한다. 최소한 아래 규칙이 필요하다.

- 입력/출력 명세서에 버전 필드를 둔다
- 계약 변경 시 팀 전체 리뷰를 거친다
- 하위 호환이 깨지는 변경은 명시적으로 공지한다
- 샘플 파일 3개 이상을 계약 문서에 함께 고정한다
- 명세서에 맞지 않는 입력은 조용히 보정하지 말고 명시적으로 실패시킨다
- 개인정보 마스킹 책임 주체를 계약서에 명시한다

즉, "포맷이 조금 달라도 웬만하면 돌게"보다 "합의된 포맷만 받는다"가 운영 안정성에 더 중요하다.

### 3-5. 민감정보 처리 계약

팀 연동이 시작되면 팀원1과의 계약 문서에는 아래 중 최소 하나가 반드시 들어가야 한다.

- STT 전처리 단계에서 PII 마스킹을 완료해서 v2에 전달한다
- 또는 v2 앞단에서 1차 정규식 기반 마스킹을 수행한다

권장 방향은 upstream에서 먼저 마스킹하고, v2에서 한 번 더 방어적으로 검증하는 이중 안전장치다.

## 4. v2 기준 최종 수정 필요사항

## 4-1. `application/chunk_processor.py`

### 수정 필요

- 입력 포맷 검증을 "묵살"이 아니라 "명시적 실패 + 원인 기록" 구조로 바꿔야 한다.
- 파싱 실패 라인을 이전 발화에 무조건 병합하는 현재 방식은 최소한 옵션화해야 한다.
- timestamp 정렬/역전 여부를 검증해야 한다.
- `HH:MM:SS` 원본 정밀도를 유지해야 한다.
- 청크 경계 탐색의 시간 복잡도와 품질 기준을 재검토해야 한다.

### 이유

- upstream 포맷 변경에 매우 약함
- 잘못 붙은 텍스트가 evidence 검증까지 오염시킴
- 초 단위 정보 손실은 디버깅과 근거 추적성을 낮춤

## 4-2. `application/analyzer_service.py`

### 수정 필요

- 실패 청크를 정상 결과처럼 집계하지 않도록 상태 모델을 분리해야 한다.
- fallback 결과를 쓰더라도 `is_fallback`, `failure_reason`, `retry_count` 같은 메타데이터를 함께 남겨야 한다.
- `lecture_id`와 `run_id`를 분리해야 한다.
- 재시작 시 checkpoint 재사용 정책을 명확히 해야 한다.
- async 경로에서 저장소 호출이 event loop를 블로킹하지 않도록 구조를 바꿔야 한다.

### 이유

- 현재 구조는 장애를 숨김
- 재실행/복구가 불안정함
- 비동기 이점이 실제로 살아나지 않음

## 4-3. `infrastructure/llm/openai_adapter.py`

### 수정 필요

- Timeout, 429, 5xx, 인증 실패, quota 초과를 분리해 처리해야 한다.
- Exponential Backoff 정책을 문서와 코드에서 동일하게 유지해야 한다.
- Refusal을 별도 상태로 분리해야 한다.
- response 파싱 실패와 evidence 검증 실패를 동일 레벨 오류로 뭉개지 말아야 한다.
- 평가용 모델과 집계용 모델의 계약을 분리하고, 결과 품질 검증 기준도 따로 가져가야 한다.
- 호출별 사용 토큰과 비용 추정치를 수집 가능하도록 결과 메타데이터를 노출해야 한다.

### 이유

- LLM 장애는 정상 흐름이 아니라 상수처럼 발생함
- refusal과 timeout은 대응 전략이 달라야 함
- 집계용 호출 실패가 최종 보고서 품질을 조용히 망칠 수 있음
- 비용 최적화와 재시도 정책 튜닝을 하려면 토큰 관측이 필수임

## 4-4. `application/validation.py`

### 수정 필요

- v2 표준 문서에서 strict substring 단독 검증 표현을 제거해야 한다.
- fuzzy matching 허용 범위와 threshold를 문서화해야 한다.
- 단순 문자열 유사도 외에 speaker/timestamp 문맥 검증 가능성을 검토해야 한다.
- false positive, false negative 기준을 샘플 데이터로 튜닝해야 한다.

### 이유

- evidence 검증이 너무 엄격하면 정답도 실패 처리됨
- 너무 느슨하면 환각 근거를 통과시킴
- 현재 영역은 정확도보다 calibration이 더 중요함

## 4-5. `application/aggregator.py`

### 수정 필요

- 실패 청크와 정상 청크를 같은 가중치로 평균 내지 않도록 해야 한다.
- 누락 점수, null 점수, evidence 없는 점수에 대한 정책을 코드와 문서에 같이 명시해야 한다.
- 정확히 10개를 보장할지, 범위를 허용할지 계약을 하나로 통일해야 한다.
- 향후 NLP 평가 결과와 LLM 평가 결과가 함께 들어올 때 병합 정책을 설계해야 한다.
- 최종 결과에 토큰 사용량과 비용 메타데이터를 함께 합산해 남겨야 한다.
- 최종 결과에 `prompt_version`, `model`, `run_id` 같은 추적용 메타데이터를 포함해야 한다.

### 이유

- 항목 누락 시 집계 왜곡 가능
- 결과 길이 계약 불일치
- LLM/NLP 결과 충돌 시 현재는 기준이 없음
- 비용과 품질 회귀를 강의 단위로 비교할 수 있어야 함

## 4-6. `infrastructure/persistence/json_repo.py`

### 수정 필요

- 현재 JSON 파일 전체 재쓰기 구조는 운영용이 아니라 임시용이라는 점을 명확히 해야 한다.
- 최소한 atomic write(temp 파일 후 rename)를 적용해야 한다.
- 파일 손상 시 조용히 초기화하지 말고 명시적 오류 및 복구 전략을 가져야 한다.
- 규모가 커지면 SQLite 같은 경량 저장소로 전환하는 계획을 문서화해야 한다.

### 이유

- 체크포인트 손상 시 진행 상황 유실
- concurrency 확장성 부족
- 비동기 처리 병목

## 4-7. `entrypoints/batch_processor.py`

### 수정 필요

- `print()` 중심 흐름을 중단하고 `logging` 기반으로 통일해야 한다.
- 배치 실행 단위의 요약 로그와 파일별 상세 로그를 분리해야 한다.
- 실행 파라미터, 시작 시각, 종료 시각, 성공/실패 청크 수, retry 횟수를 남겨야 한다.
- import 시점 event loop policy 변경은 영향을 검토한 뒤 최소화해야 한다.
- 배치 단위 총 토큰, 총 비용, 강의별 평균 비용을 요약 메타데이터로 남겨야 한다.

### 이유

- 운영 장애 추적이 어려움
- 배치 로그는 사후 분석 수단이므로 구조화가 필수
- 배치가 비정상적으로 비싸게 돌았는지 즉시 알 수 있어야 함

## 4-8. `application/prompts.py`

### 수정 필요

- 프롬프트를 문자열 상수로만 두지 말고 버전 식별자와 함께 관리해야 한다.
- 최종 결과 JSON 메타데이터에 현재 사용된 `prompt_version`을 남겨야 한다.
- 프롬프트 변경 시 어떤 평가 기준이 바뀌었는지 changelog 수준의 기록을 유지해야 한다.

### 이유

- 프롬프트는 코드와 동일한 수준의 형상 관리 대상임
- 품질 회귀가 생겼을 때 prompt 영향인지 code 영향인지 구분해야 함

## 4-9. 입력 전처리/보안 필터 계층

### 수정 필요

- 외부 LLM 호출 전에 최소한 1차 PII/민감정보 필터를 둘지 결정해야 한다.
- upstream 마스킹이 계약상 보장되더라도, v2에서 방어적 검사를 한 번 더 하는 것이 안전하다.
- 정규식 기반 1차 필터라도 두어 전화번호, 이메일, 주민식별성 문자열 등은 마스킹할 수 있어야 한다.

### 이유

- 기능 버그보다 보안 사고가 더 큰 장애가 될 수 있음
- upstream 계약만 믿고 그대로 외부로 보내는 것은 위험함

## 5. 프롬프트/평가 설계 측면의 수정 필요사항

### 5-1. 13개 항목 일괄 평가를 재검토

다음 세 가지 안 중 하나로 정리해야 한다.

1. 13개를 유지하되 항목별 evidence 강제와 사후 검증 강화
2. 13개를 3~4개 그룹으로 나눠 다단계 평가
3. LLM 평가 항목과 NLP 평가 항목을 명확히 분리

현재 상태에서 가장 위험한 것은 "한 번에 다 시키고, 결과가 나오면 일단 믿는 구조"다.

### 5-2. LLM vs NLP 경계 명확화

v2 기준으로 문서에 아래를 명시해야 한다.

- 어떤 항목은 LLM 담당
- 어떤 항목은 NLP 담당
- 동일 항목을 둘 다 평가하면 어떤 쪽을 우선할지
- 충돌 시 어떤 규칙으로 병합할지

### 5-3. 근거 없는 장점/이슈 생성을 금지하는 정책 강화

프롬프트에 이미 방향이 있어도, 코드 레벨 검증과 결과 QA 규칙이 함께 있어야 한다.

- evidence 없는 high/low score를 허용할지
- evidence가 없으면 자동 3점으로 둘지
- 특정 항목은 evidence 필수로 할지

이 정책이 없으면 모델 출력의 일관성이 유지되지 않는다.

## 6. Resilience 요구사항

아래는 "있으면 좋다"가 아니라 배포 전 체크리스트다.

### 필수

- Timeout 재시도
- 429 재시도
- 5xx 재시도
- 인증 실패 즉시 중단
- quota 초과 즉시 중단
- refusal 별도 처리
- 구조화 출력 파싱 실패 별도 처리
- evidence 검증 실패 별도 처리

### 재시도 정책

- Exponential Backoff
- jitter 포함
- 최대 재시도 횟수 고정
- retry 가능한 오류와 불가능한 오류 구분

### 결과 상태

최종 청크 결과 모델은 최소한 아래 상태를 구분해야 한다.

- `SUCCESS`
- `FAILED_RETRYABLE`
- `FAILED_NON_RETRYABLE`
- `REFUSED`
- `FALLBACK_USED`

### 비용/토큰 메타데이터

최종 결과와 배치 결과에는 아래 메타데이터가 남아야 한다.

- prompt_version
- model
- total_prompt_tokens
- total_completion_tokens
- total_tokens
- estimated_cost
- llm_call_count

이 항목들은 로깅으로만 끝내지 말고 JSON 결과 메타데이터로도 남겨야 한다.

### Graceful Degradation 정책

v2는 성공률 100%를 가정하지 말고, 실패 시 우아하게 저하되는 경로를 설계해야 한다.

- 청크 단위 실패가 전체 배치를 죽이지 않도록 한다
- 실패 청크는 정상 청크와 구분되는 상태와 메타데이터를 남긴다
- fallback을 쓰더라도 downstream이 이를 식별할 수 있어야 한다
- 집계 결과는 "정상 분석"과 "부분 실패 포함 분석"을 구분해 표시해야 한다
- 재시도 한도 초과 후에도 운영자가 원인을 추적할 수 있어야 한다

핵심은 "멈추지 않는 것"이 아니라 "어떤 품질로 계속 갔는지 투명하게 남기는 것"이다.

## 7. Logging 고도화 요구사항

v2는 `print()` 중심 디버깅에서 벗어나야 한다. 최소 요구사항은 다음과 같다.

### 로그 레벨

- DEBUG
- INFO
- WARNING
- ERROR

### 로그 키

- file_name
- lecture_id
- run_id
- chunk_id
- stage
- model
- retry_count
- elapsed_ms
- error_type
- error_message
- prompt_tokens
- completion_tokens
- total_tokens
- estimated_cost
- prompt_version

### 로그 예시

```text
[ERROR] file_name=2026-02-25_xxx.txt lecture_id=260225_kdt-backendj-21th run_id=02 chunk_id=7 stage=llm_call error_type=RateLimitError retry_count=2 message="HTTP 429"
```

### 파일 로그 요구사항

- 배치 전체 로그
- 파일별 처리 로그
- 실패 전용 로그

### 구현 원칙

- 런타임 경로에서 `print()`를 제거하고 `logging`으로 통일한다
- 파싱 실패, LLM 실패, 저장 실패를 서로 다른 stage로 남긴다
- 파일명과 청크 ID가 없는 에러 로그는 금지한다
- 운영 로그와 개발용 디버그 로그를 분리한다
- 비용 이상 징후를 감지할 수 있도록 토큰/비용 필드를 로그에 포함한다

## 8. 테스트 및 보완 계획

현재 v2는 "정상 동작"보다 "예외 상황"을 검증해야 한다. 우선순위는 아래와 같다.

### 8-1. 입력 파싱 테스트

- 정상 포맷
- timestamp 누락
- speaker_id 누락
- 줄바꿈 깨짐
- 여러 줄 발화
- 역순 timestamp

### 8-2. LLM 장애 테스트

- timeout 모의
- 429 모의
- 500 모의
- refusal 모의
- malformed structured output 모의

### 8-3. evidence 검증 테스트

- 완전 일치
- 띄어쓰기만 다른 경우
- 조사만 다른 경우
- 일부 축약된 경우
- 실제 환각 문장

### 8-4. aggregator 테스트

- 누락 항목 포함
- null 점수 포함
- evidence 없는 청크 포함
- 실패 청크 포함
- 정상/실패 혼합 배치

### 8-5. checkpoint/재시작 테스트

- 중간 종료 후 재실행
- checkpoint 손상
- 일부 청크만 성공한 상태
- 같은 lecture 재분석

### 8-6. 실데이터 우선 통합 테스트

mock 데이터는 보조 수단일 뿐, 안정성 검증의 기준이 되어서는 안 된다. v2는 노이즈가 있는 실제 STT 결과물을 기준으로 먼저 검증해야 한다.

- 현재 사용 중인 `dataset/강의 스크립트/*.txt`를 임시 입력 표준으로 간주하고 parser 기준과 대조한다
- 현재 parser가 기대하는 형식과 실제 샘플을 바로 대조한다
- `dataset/강의 스크립트/*.txt` 같은 실데이터 1건으로 end-to-end를 먼저 돌린다
- mock 데이터는 회귀 테스트 용도로만 유지한다

실무에서 터지는 에러는 대개 mock이 아니라 실데이터에서 나온다. 따라서 v2의 예외 처리 우선순위도 실데이터 기준으로 잡아야 한다.

### 8-7. 골든 데이터셋 기반 회귀 테스트

샘플 3개는 스모크 테스트에는 유용하지만, 품질 보증용으로는 부족하다. v2는 코드와 프롬프트 변경 시 비교 기준이 되는 고정 골든 데이터셋을 가져야 한다.

- 정상적인 강의
- 노이즈가 심한 강의
- 잡담이 많은 강의
- 질의응답이 많은 강의
- 욕설/비정형 발화가 섞인 강의
- 극단적으로 짧거나 긴 청크가 나오는 강의

권장 규모는 10~20개 수준의 고정 텍스트 셋이다. 코드를 수정하거나 프롬프트를 바꿀 때마다 이 셋을 돌려 이전 결과와 비교해야 한다.

- 점수 분포 변화
- evidence 통과율 변화
- refusal/실패 비율 변화
- 토큰 사용량 변화
- 총 비용 변화

이 회귀 테스트가 없으면 퍼지 매칭 도입, 프롬프트 수정, retry 정책 변경이 실제로 개선인지 퇴행인지 판단할 수 없다.

참고:

- 현재 `ik/LLMEngine` 하위에서 자동 테스트 자산은 충분히 보이지 않는다.
- v2는 코드 수정과 함께 테스트 자산도 반드시 같이 보강되어야 한다.

## 9. 우선순위별 정리

## P0. 배포 전 반드시 수정

- 실패 청크를 정상 결과처럼 집계하지 않기
- lecture_id / run_id / checkpoint 정책 통일
- 팀 간 데이터 교환 포맷 계약 문서 확정
- 계약 문서 변경 시 리뷰/버전 관리 규칙 확정
- PII 마스킹 책임 주체와 처리 위치 확정
- logging 전면 정비
- retry/refusal/failure 상태 모델 분리
- strict substring 단독 검증 정책 폐기
- "100% 성공" 목표 제거 및 graceful degradation 정책 수립
- prompt_version, token usage, cost metadata 남기기

## P1. 빠르게 보완

- 13개 항목 평가 전략 재설계
- aggregator의 누락/충돌/혼합 결과 방어 로직 강화
- checkpoint 저장소 atomic write 적용
- async 경로에서 저장소 병목 제거
- 골든 데이터셋 기반 회귀 테스트 구축

## P2. 중기 개선

- JSON checkpoint에서 SQLite 등으로 전환 검토
- LLM/NLP 병합 전용 integrator 계층 도입
- 운영 메타데이터 대시보드화
- 데이터 및 로그 보존 기한(retention policy) 수립

예시 정책:

- 생성 후 30일이 지난 원본 STT 스크립트, 청크 JSON, 로그 파일은 자동 압축 또는 삭제
- 디스크 사용량 임계치 도달 전 알림 또는 정리 작업 수행
- checkpoint와 최종 산출물은 보존 등급을 분리해 관리

## 11. 1차 작업 목록

1. 현재 사용 중인 `dataset/강의 스크립트/*.txt`를 임시 입력 표준(working contract)으로 선언하고, 그 실제 형식을 기준으로 `입력 Schema Contract` 초안을 먼저 고정한다.
2. `ChunkResult`와 최종 summary 메타데이터에 `status`, `is_fallback`, `failure_reason`, `retry_count`, `prompt_version`, `token/cost` 필드를 어떻게 남길지 데이터 모델을 먼저 정리한다.
3. `entrypoints/batch_processor.py`와 공통 실행 경로의 `print()`를 `logging`으로 교체하고, `file_name`, `chunk_id`, `stage`, `error_type`, `token/cost`가 포함된 로그 포맷을 통일한다.
4. `infrastructure/llm/openai_adapter.py`에서 Timeout, 429, 5xx, refusal, non-retryable 오류를 분리하고, 호출별 token usage와 estimated cost를 수집하도록 바꾼다.
5. `application/analyzer_service.py`에서 실패 청크를 정상 결과처럼 섞지 않도록 상태 모델과 graceful degradation 경로를 반영한다.
6. `application/validation.py`에서 evidence 검증 정책을 strict substring 중심에서 fuzzy matching 기준 중심으로 문서와 코드 모두 정리한다.
7. `application/aggregator.py`에서 실패 청크 제외/감산 정책, 누락 점수 처리 정책, 최종 메타데이터 합산 정책을 확정한다.
8. 실제 STT 파일 1건으로 end-to-end를 돌려 파싱 실패, refusal, 재시도, 비용, 로그, 최종 JSON 메타데이터가 의도대로 남는지 확인한다.
9. 골든 데이터셋 10~20건을 정의하고, 이후 프롬프트/검증 로직 변경 시 회귀 테스트 기준으로 삼는다.

이 1차 작업의 목표는 "품질 개선"이 아니라 "깨져도 추적 가능하고, 실패해도 복구 가능한 v2"를 만드는 것이다.

## 12. 최종 결론

v2는 "구조는 잡혀 있으나 운영 신뢰성은 아직 부족한 상태"다. 현재 가장 위험한 부분은 모델 성능 자체보다 다음 세 가지다.

- 실패 은폐
- 인터페이스 계약 부재
- 복구 불가능한 배치 구조

따라서 지금 당장 해야 할 일은 프롬프트 미세조정이 아니다. 먼저 아래 순서로 정리해야 한다.

1. 팀 간 데이터 계약 확정
2. 실패/재시도/복구 상태 모델 확정
3. checkpoint와 run 정책 확정
4. logging/테스트 체계 확정
5. 그 다음에 평가 품질과 프롬프트 최적화

이 순서를 지키지 않으면, v2는 결과가 좋아 보여도 실제 운영에서는 장애 원인을 숨기고 비용만 키우는 시스템이 될 가능성이 높다.
