# LLM 분석 엔진 테스트 명세서

본 문서는 강의 품질 평가 체계 중 **LLM 분석 엔진이 항목별 점수와 종합 해석을 생성하는 과정**에 대한 테스트 명세를 정의한다.

- 대상 엔진: LLM 분석 엔진
- 대상 산출물: 항목별 점수, 강점/이슈/근거, 카테고리별 요약 점수
- 점수 체계: 5점(매우 우수) ~ 1점(매우 미흡)
- 관련 출력 예시

```json
{
  "llm_aggregated_analysis": {
    "summary_scores": {
      "lecture_structure": {
        "learning_objective_intro": 3.8,
        "previous_lesson_linkage": 3.1,
        "explanation_sequence": 4.0,
        "key_point_emphasis": 3.4,
        "closing_summary": 2.7
      },
      "concept_clarity": {
        "concept_definition": 4.1,
        "analogy_example_usage": 3.6,
        "prerequisite_check": 3.3
      },
      "practice_linkage": {
        "example_appropriateness": 4.0,
        "practice_transition": 3.5,
        "error_handling": 3.2
      },
      "interaction": {
        "participation_induction": 2.8,
        "question_response_sufficiency": 3.0
      }
    },
    "overall_strengths": [],
    "overall_issues": [],
    "overall_evidences": []
  }
}
```

---

## 1. 문서 목적

본 명세서는 다음을 검증하기 위해 사용한다.

1. LLM 엔진이 `rubric_spec.md`의 항목 정의와 점수 rubric을 일관되게 반영하는지 검증한다.
2. 입력 스크립트와 NLP feature를 바탕으로 항목별 점수를 안정적으로 산출하는지 검증한다.
3. 강점, 이슈, 근거가 점수와 정합적으로 생성되는지 검증한다.
4. 동일 입력에 대한 재실행 결과가 허용 가능한 편차 내에서 유지되는지 검증한다.
5. 항목별 출력이 최종 집계 구조(`summary_scores`, `overall_strengths`, `overall_issues`, `overall_evidences`)에 정상 반영되는지 검증한다.

---

## 2. 테스트 범위

### 2.1 직접 검증 대상 항목

아래 항목은 LLM 엔진이 직접 판단하거나 NLP feature와 결합하여 해석해야 한다.

| 카테고리 | 평가 항목 | 항목 ID |
|---|---|---|
| 강의 도입 및 구조 | 학습 목표 안내 | `learning_objective_intro` |
| 강의 도입 및 구조 | 전날 복습 연계 | `previous_lesson_linkage` |
| 강의 도입 및 구조 | 설명 순서 | `explanation_sequence` |
| 강의 도입 및 구조 | 핵심 내용 강조 | `key_point_emphasis` |
| 강의 도입 및 구조 | 마무리 요약 | `closing_summary` |
| 개념 설명 명확성 | 개념 정의 | `concept_definition` |
| 개념 설명 명확성 | 비유 및 예시 활용 | `analogy_example_usage` |
| 개념 설명 명확성 | 선행 개념 확인 | `prerequisite_check` |
| 예시 및 실습 연계 | 예시 적절성 | `example_appropriateness` |
| 예시 및 실습 연계 | 실습 연계 | `practice_transition` |
| 예시 및 실습 연계 | 오류 대응 | `error_handling` |
| 수강생 상호작용 | 참여 유도 | `participation_induction` |
| 수강생 상호작용 | 질문 응답 충분성 | `question_response_sufficiency` |

### 2.2 NLP 보조 입력 활용 항목

아래 지표는 LLM이 직접 점수화하지 않더라도 판정 보조에 활용할 수 있다.

- `language_quality.repeat_expressions`
- `language_quality.repeat_ratio`
- `language_quality.incomplete_sentence_ratio`
- `language_quality.speech_style_ratio`
- `concept_clarity_metrics.speech_rate_wpm`
- `interaction_metrics.understanding_question_count`

본 문서에서는 위 지표를 **보조 feature**로 취급하며, LLM이 이를 무시하거나 과도하게 의존하지 않는지도 함께 검증한다.

## 3. 공통 입력/출력 규격

### 3.1 입력 단위

LLM 엔진 입력은 원칙적으로 다음을 포함한다.

- 강의 원문 스크립트 또는 발췌 구간
- NLP 분석 엔진 출력 JSON
- 평가 대상 항목 ID 또는 전체 평가 요청
- 평가 기준 및 점수 rubric

### 3.2 공통 전제

- 모든 테스트는 동일한 rubric 버전을 사용해야 한다.
- 점수는 1~5 범위 내에서 산출되어야 한다.
- 소수점 표기 자릿수는 사전 정의한 정책을 따른다.
- 전체 강의 평가와 발췌 구간 평가를 명확히 구분한다.
- 항목 점수와 서술형 근거는 서로 모순되지 않아야 한다.

### 3.3 공통 검증 항목

각 테스트 케이스는 최소한 아래 항목을 검증한다.

- 항목 점수 범위 적합성
- rubric 분기와의 정합성
- 근거 문장 및 해석의 타당성
- NLP feature 활용의 적절성
- 재실행 시 결과 안정성
- 최종 집계 구조 반영 여부

---

## 4. 테스트 케이스 형식

각 테스트 케이스는 아래 형식으로 작성한다.

```md
## TC-LLM-XXX 테스트명

- 대상 항목:
- 목적:
- 입력 데이터:
- 전처리/가정:
- 검증 절차:
- 기대 결과:
- 실패 조건:
- 비고:
```

---

## 5. 테스트 유형 정의

각 항목에 대해 최소 아래 세 유형을 작성한다.

1. **positive**: rubric 상 4~5점이 기대되는 사례
2. **borderline**: rubric 상 3점 전후가 기대되는 사례
3. **negative**: rubric 상 1~2점이 기대되는 사례

필요 시 아래 유형을 추가한다.

- **conflict**: 스크립트와 NLP feature가 일부 상충하는 사례
- **aggregation**: 개별 항목이 전체 강점/이슈 문장으로 적절히 반영되는 사례
- **robustness**: 길이, 표현 다양성, 발췌 범위 차이에도 판단이 유지되는 사례

## 6. 항목별 테스트 명세

# 6.1 학습 목표 안내

- **항목 ID**: `learning_objective_intro`
- **관련 JSON 경로**: `summary_scores.lecture_structure.learning_objective_intro`
- **관련 rubric 항목**: 강의 시작 시 오늘의 학습 목표와 진행 순서를 명확히 안내하는가.

### 검증 포인트

1. 강의 시작 구간에서 학습 목표와 진행 순서를 모두 인식하는가.
2. 목표만 있고 순서가 없을 때 3점대 이하로 제한하는가.
3. 단순 도입 멘트와 실제 목표 안내를 구분하는가.
4. 관련 근거가 강의 초반 구간과 정합적으로 연결되는가.

### 대표 테스트 케이스

#### TC-LLM-001 목표와 진행 순서가 모두 명확한 도입 구간
- **대상 항목**: `learning_objective_intro`
- **목적**: 학습 목표와 진행 순서가 모두 직접적으로 제시된 경우 고득점이 산출되는지 검증한다.
- **입력 데이터**:
  - “오늘은 트랜잭션의 개념을 먼저 정리하고, 그다음 전파 속성을 본 뒤 마지막으로 실습으로 확인하겠습니다.”
- **기대 결과**:
  - 점수가 4~5점 범위에 위치한다.
  - 근거에 목표와 순서 표현이 모두 반영된다.
- **실패 조건**:
  - 3점 이하 산출
  - 단순 주제 소개로만 해석

#### TC-LLM-002 목표만 제시되고 순서 안내가 없는 도입 구간
- **대상 항목**: `learning_objective_intro`
- **목적**: 한 요소만 분명한 경우 중간 점수대가 산출되는지 검증한다.
- **입력 데이터**:
  - “오늘은 트랜잭션 전파 속성을 배우겠습니다.”
- **기대 결과**:
  - 점수가 3점 전후 또는 3~4점 하단에 위치한다.
- **실패 조건**:
  - 5점에 가깝게 산출
  - 진행 순서를 임의로 추론함

#### TC-LLM-003 도입 안내 없이 곧바로 본론에 진입하는 구간
- **대상 항목**: `learning_objective_intro`
- **목적**: 목표 안내 부재 시 저득점이 산출되는지 검증한다.
- **입력 데이터**:
  - “자, 그럼 코드부터 보겠습니다. 여기 서비스 레이어를 열어보세요.”
- **기대 결과**:
  - 점수가 1~2점 범위에 위치한다.
- **실패 조건**:
  - 목표 안내가 있다고 오판

3. 복습과 연결 중 하나만 존재할 때 중간 점수대가 산출되는가.

### 대표 테스트 케이스

- **실패 조건**: 복습만 인식하거나 연결만 인식

- **실패 조건**: 4점 이상 과대평가

#### TC-LLM-006 이전 수업 언급이 전혀 없는 구간

- **실패 조건**: 암묵적 연결을 과도하게 추론

3. 발췌 구간만 보고 전체 구조를 과대추정하지 않는가.

- **실패 조건**: 세 요소를 분리 인식하지 못함

- **기대 결과**: 2~3점 또는 3점대 초반
- **실패 조건**: 구조적 전개로 과대평가

#### TC-LLM-009 설명 순서가 뒤섞인 산만한 구간
- **기대 결과**: 1~2점

- **기대 결과**: 2~3점 또는 3점대 초반

## 7. 공통 품질 검증 테스트

### TC-LLM-040 출력 스키마 유효성 검증
- **목적**: 전체 LLM 출력 JSON 구조가 사전 정의된 스키마를 준수하는지 검증한다.
- **검증 항목**:
  - `llm_aggregated_analysis` 존재 여부
  - `summary_scores` 하위 카테고리 존재 여부
  - 점수 필드의 숫자형 여부
  - `overall_strengths`, `overall_issues`, `overall_evidences`의 구조 적합성
- **실패 조건**:
  - 필드 누락
  - 타입 불일치
  - 정의되지 않은 카테고리 출력

### TC-LLM-041 항목 점수와 서술형 근거 정합성 검증
- **목적**: 항목별 점수와 overall 문장이 서로 모순되지 않는지 검증한다.
- **검증 항목**:
  - 낮은 점수 항목이 강점으로 들어가지 않는지
  - 높은 점수 항목이 이슈로 들어가지 않는지
  - evidences가 실제 스크립트와 연결되는지
- **실패 조건**:
  - 점수-요약 간 모순
  - 근거 문장 부재 또는 왜곡

### TC-LLM-042 동일 입력 재실행 안정성 검증
- **목적**: 동일 입력에 대해 점수와 서술형 판단이 허용 가능한 편차 내에서 유지되는지 검증한다.
- **검증 항목**:
  - 동일 입력 반복 실행 시 점수 변동 폭
  - strength/issue 핵심 문장의 의미 일관성
- **실패 조건**:
  - 점수 편차 과도
  - 핵심 판단 방향 역전

### TC-LLM-043 NLP feature 활용 균형성 검증
- **목적**: LLM이 보조 feature를 참고하되 스크립트 본문과 모순되는 과잉 의존을 하지 않는지 검증한다.
- **입력 데이터**:
  - 스크립트상 구조가 양호하나 일부 NLP 수치가 경계값인 사례
  - 스크립트상 문제점이 뚜렷하나 NLP 수치가 평균적인 사례
- **기대 결과**:
  - 점수 판단이 본문 내용과 feature를 균형 있게 반영한다.
- **실패 조건**:
  - feature만 보고 점수를 결정
  - feature를 완전히 무시

### TC-LLM-044 부분 발췌와 전체 강의 평가 구분 검증
- **목적**: 발췌 구간 입력과 전체 강의 입력을 구분하여 과도한 일반화를 하지 않는지 검증한다.
- **검증 항목**:
  - 도입 항목은 도입 발췌에서만 평가 근거가 충분한지 확인
  - 마무리 항목은 후반부 발췌 없이는 고평가를 자제하는지 확인
- **실패 조건**:
  - 부분 구간만 보고 전체 구조를 단정

- negative 케이스 2건 이상

## 9. 작성 및 운영 원칙

1. 기대 결과는 가능하면 **정확한 점수**보다 **허용 범위**로 적는다.
2. LLM 테스트는 스크립트만 보는 케이스와 NLP feature를 함께 보는 케이스를 분리한다.
3. 부분 발췌 테스트는 평가 가능한 항목과 불가능한 항목을 구분해서 설계한다.
4. rubric 문구가 변경되면 본 문서와 테스트 케이스, 회귀 기준을 동시에 갱신한다.
5. 프롬프트 변경, 모델 변경, temperature 변경 시 회귀 테스트를 반드시 재실행한다.
6. 점수뿐 아니라 strength/issue/evidence의 질적 일관성도 함께 검토한다.
