# NLP 분석 엔진 테스트 명세서

본 문서는 강의 품질 평가 체계 중 **NLP 분석 엔진이 직접 산출하거나 1차적으로 판정 보조에 활용하는 지표**에 대한 테스트 명세를 정의한다.

- 대상 엔진: NLP 분석 엔진
- 대상 산출물: 전처리 결과 및 NLP feature JSON
- 점수화 방식: 본 문서는 직접 점수를 부여하지 않으며, **지표 산출 정확성**과 **후속 LLM 평가 입력으로서의 신뢰성**을 검증한다.
- 관련 출력 예시

```json
{
  "lecture_id": "2026-02-02_kdt-backendj-21th",
  "language_quality": {
    "repeat_expressions": {
      "이제": 42,
      "그래서": 36,
      "어쨌든": 11
    },
    "repeat_ratio": 0.18,
    "incomplete_sentence_ratio": 0.05,
    "speech_style_ratio": {
      "formal": 0.92,
      "informal": 0.08
    }
  },
  "concept_clarity_metrics": {
    "speech_rate_wpm": 176
  },
  "interaction_metrics": {
    "understanding_question_count": 21
  }
}
```

---

## 1. 문서 목적

본 명세서는 다음을 검증하기 위해 사용한다.

1. 전처리 및 NLP 엔진이 정의된 feature를 일관된 형식으로 산출하는지 검증한다.
2. 반복 표현, 문장 완결성, 화법 일관성, 발화 속도, 이해 확인 질문 수와 같이 **수치화 가능한 항목**이 기대값에 맞게 계산되는지 검증한다.
3. 후속 LLM 엔진 입력으로 전달되는 값이 누락, 왜곡, 단위 불일치 없이 생성되는지 검증한다.

---

## 2. 테스트 범위

### 2.1 직접 검증 대상 항목

아래 항목은 NLP 엔진이 직접 산출하거나 직접적인 수치 근거를 제공해야 한다.

| 평가 항목 | 항목 ID | 관련 JSON 경로 |
|---|---|---|
| 불필요한 반복 표현 | `repeat_expressions` | `language_quality.repeat_expressions`, `language_quality.repeat_ratio` |
| 발화 완결성 | `utterance_completeness` | `language_quality.incomplete_sentence_ratio` |
| 언어 일관성 | `speech_style_consistency` | `language_quality.speech_style_ratio` |
| 발화 속도 적절성 | `speech_rate_appropriateness` | `concept_clarity_metrics.speech_rate_wpm` |
| 이해 확인 질문 | `understanding_question` | `interaction_metrics.understanding_question_count` |

### 2.2 간접 지원 항목

아래 항목은 NLP 엔진이 직접 점수화하지 않지만, 추후 LLM 판단에 보조 근거로 활용될 수 있다.

- 핵심 내용 강조
- 참여 유도
- 질문 응답 충분성
- 설명 순서

본 문서에서는 위 항목을 직접 테스트 대상으로 삼지 않는다.

---

## 3. 공통 입력/출력 규격

### 3.1 입력 단위

NLP 엔진 입력은 원칙적으로 다음 중 하나 이상을 포함한다.

- 강의 원문 스크립트
- 발화 단위 분리 텍스트
- 타임스탬프 포함 자막/전사 데이터
- 화자 구분 정보(있는 경우)

### 3.2 공통 전제

- 모든 테스트는 **동일한 전처리 규칙**을 사용해야 한다.
- 띄어쓰기, 구두점, 줄바꿈 처리 방식은 테스트 수행 전 고정한다.
- 타임스탬프 기반 지표는 초 단위 또는 밀리초 단위 중 하나로 통일한다.
- 질문 수 계산 시 **수사적 표현**, **자문자답**, **실제 수강생 이해 확인 질문**의 구분 기준을 사전에 고정한다.

### 3.3 공통 검증 항목

각 테스트 케이스는 최소한 아래 항목을 검증한다.

- JSON 필드 존재 여부
- 필드 타입 적합성
- 값의 범위 유효성
- 계산 방식 일관성
- 경계값 처리 정확성

---

## 4. 테스트 케이스 형식

각 테스트 케이스는 아래 형식으로 작성한다.

```md
## TC-NLP-XXX 테스트명

- 대상 항목:
- 목적:
- 입력 데이터:
- 전처리/가정:
- 검증 절차:
- 기대 결과:
- 실패 조건:
- 비고:
```

## 5. 항목별 테스트 명세

# 5.1 불필요한 반복 표현

- **항목 ID**: `repeat_expressions`
- **관련 JSON 경로**:
  - `language_quality.repeat_expressions`
  - `language_quality.repeat_ratio`
- **관련 rubric 항목**: 동일 단어·문장 및 “이제”, “그래서” 등 특정 표현을 과도하게 반복하지 않는가.

### 검증 포인트

1. 반복 표현 사전이 정상적으로 추출되는가.
2. 동일 표현의 빈도가 정확히 집계되는가.
3. `repeat_ratio`가 정의된 계산식에 맞게 산출되는가.
4. 불용 반복 표현과 내용어 반복을 동일하게 처리하지 않는가.
5. 띄어쓰기, 문장부호, 어미 변형에 의해 동일 표현 집계가 과도하게 분산되지 않는가.

### 대표 테스트 케이스

#### TC-NLP-001 반복 표현이 거의 없는 일반 강의 발화
- **대상 항목**: `repeat_expressions`
- **목적**: 반복 표현이 적은 텍스트에서 낮은 반복 비율이 산출되는지 검증한다.
- **입력 데이터**:
  - 주제 전개는 유지하되 동일 접속 부사 반복이 거의 없는 강의 발화
- **전처리/가정**:
  - 조사와 어미 차이는 동일 표현 판정에서 제외하지 않는다.
- **검증 절차**:
  1. 텍스트를 입력한다.
  2. 반복 표현 사전과 반복 비율을 산출한다.
  3. 상위 반복 표현 수와 전체 반복 비율을 확인한다.
- **기대 결과**:
  - `repeat_expressions`가 비어 있거나 소수 항목만 포함한다.
  - `repeat_ratio`가 낮은 값으로 산출된다.
- **실패 조건**:
  - 의미 없는 중복 검출이 다수 발생한다.
  - 실제 반복이 적음에도 `repeat_ratio`가 과도하게 높다.

#### TC-NLP-002 특정 표현이 비정상적으로 자주 반복되는 발화
- **대상 항목**: `repeat_expressions`
- **목적**: 동일 표현 다빈도 사용이 정확히 집계되는지 검증한다.
- **입력 데이터**:
  - “이제”, “그래서”, “어쨌든”이 반복적으로 등장하는 발화
- **검증 절차**:
  1. 반복 표현이 포함된 텍스트를 입력한다.
  2. 표현별 count와 `repeat_ratio`를 확인한다.
- **기대 결과**:
  - 주요 반복 표현이 사전에 포함된다.
  - 각 표현의 빈도가 수작업 집계값과 일치한다.
  - `repeat_ratio`가 TC-NLP-001보다 높다.
- **실패 조건**:
  - 반복 표현 count 누락
  - count 오집계
  - 표현 수는 맞지만 `repeat_ratio` 계산이 불일치

#### TC-NLP-003 문장부호/띄어쓰기 변형이 있는 동일 표현 처리
- **대상 항목**: `repeat_expressions`
- **목적**: “그래서”, “그래서요”, “그래서,” 등 표면형 차이 처리 기준이 일관적인지 검증한다.
- **입력 데이터**:
  - 동일 어근 표현이 다양한 형태로 반복되는 텍스트
- **전처리/가정**:
  - 어떤 수준까지 동일 표현으로 묶을지 사전 규칙을 고정한다.
- **기대 결과**:
  - 정의된 규칙에 맞게 count가 산출된다.
  - 규칙 외 케이스를 임의로 병합하지 않는다.
- **실패 조건**:
  - 실행마다 count가 달라진다.
  - 병합 기준이 일관되지 않다.

#### TC-NLP-004 짧은 텍스트 입력에서의 반복 비율 경계값
- **대상 항목**: `repeat_expressions`
- **목적**: 매우 짧은 입력에서 `repeat_ratio`가 과도하게 왜곡되지 않는지 검증한다.
- **입력 데이터**:
  - 1~3문장 수준의 짧은 발화
- **기대 결과**:
  - 분모 정의가 안정적이어야 하며 0 나누기 오류가 발생하지 않는다.
  - 짧은 길이 특성으로 인해 해석 주의가 필요한 경우라도 수치는 일관되게 산출된다.
- **실패 조건**:
  - division by zero
  - null/NaN/음수 출력

# 5.2 발화 완결성

- **항목 ID**: `utterance_completeness`
- **관련 JSON 경로**: `language_quality.incomplete_sentence_ratio`
- **관련 rubric 항목**: 문장이 완결된 형태로 끝맺음되는가(중간에 끊기지 않는가).

### 검증 포인트

1. 미완결 문장을 탐지하는 규칙이 정의되어 있는가.
2. `incomplete_sentence_ratio`가 전체 문장 수 대비 정확히 계산되는가.
3. 줄바꿈, 말줄임표, 자막 분절로 인해 정상 문장이 미완결로 오검출되지 않는가.
4. 실제 중단 발화는 누락되지 않는가.

### 대표 테스트 케이스

#### TC-NLP-005 대부분 완결된 문장으로 구성된 강의 발화
- **대상 항목**: `utterance_completeness`
- **목적**: 정상 강의 발화에서 낮은 미완결 비율이 산출되는지 검증한다.
- **입력 데이터**:
  - 종결 표현과 문장 경계가 분명한 텍스트
- **기대 결과**:
  - `incomplete_sentence_ratio`가 낮다.
  - 완결 문장이 미완결로 다수 분류되지 않는다.
- **실패 조건**:
  - 종결 문장 대량 오검출

#### TC-NLP-006 중단 발화와 자기수정이 많은 발화
- **대상 항목**: `utterance_completeness`
- **목적**: 끊긴 문장이 잦은 텍스트에서 미완결 비율이 증가하는지 검증한다.
- **입력 데이터**:
  - “여기서 이게… 아니 잠깐만요”, “그러니까 이 부분은…” 형태가 반복되는 텍스트
- **검증 절차**:
  1. 문장 단위로 수작업 정답셋을 만든다.
  2. 자동 산출 결과와 비교한다.
- **기대 결과**:
  - 미완결 문장 비율이 TC-NLP-005보다 높다.
  - 자기수정형 중단 발화가 일정 비율 이상 검출된다.
- **실패 조건**:
  - 중단 발화 누락
  - 말버릇까지 무조건 미완결로 분류

#### TC-NLP-007 자막 줄바꿈으로 인한 가짜 단문 처리
- **대상 항목**: `utterance_completeness`
- **목적**: 자막 편집상 줄바꿈이 문장 미완결로 잘못 판정되지 않는지 검증한다.
- **입력 데이터**:
  - 하나의 완결 문장이 여러 줄로 분리된 자막 텍스트
- **기대 결과**:
  - 줄바꿈만으로 미완결 비율이 상승하지 않는다.
- **실패 조건**:
  - 줄 단위 처리로 인해 미완결 문장 수가 과대 산출된다.

#### TC-NLP-008 빈 입력 또는 문장 수가 0인 입력
- **대상 항목**: `utterance_completeness`
- **목적**: 입력 결손 시 시스템이 안정적으로 동작하는지 검증한다.
- **기대 결과**:
  - 오류 없이 사전 정의된 기본값 또는 예외 처리 규칙을 따른다.
- **실패 조건**:
  - 런타임 오류
  - 비정상 수치 출력

# 5.3 언어 일관성

- **항목 ID**: `speech_style_consistency`
- **관련 JSON 경로**: `language_quality.speech_style_ratio`
- **관련 rubric 항목**: 강의 전반에 걸쳐 존댓말/반말이 일관되게 사용되는가.

### 검증 포인트

1. 존댓말과 반말을 구분하는 규칙이 명확한가.
2. `formal`, `informal` 비율 합이 1.0 또는 정의된 범위 내로 산출되는가.
3. 인용문, 코드 낭독, 채팅 읽기 등이 화법 분류를 과도하게 왜곡하지 않는가.
4. 혼합 화법 강의에서 비율 변화가 실제 텍스트와 대체로 대응하는가.

### 대표 테스트 케이스

#### TC-NLP-009 존댓말 위주의 강의 발화
- **대상 항목**: `speech_style_consistency`
- **목적**: 존댓말 중심 텍스트에서 `formal` 비율이 높게 산출되는지 검증한다.
- **입력 데이터**:
  - “보겠습니다”, “해보겠습니다”, “되셨죠” 등 존댓말 종결이 지속되는 텍스트
- **기대 결과**:
  - `formal` 비율이 높다.
  - `informal` 비율이 상대적으로 낮다.
- **실패 조건**:
  - 종결 어미를 인식하지 못해 비율이 뒤집힌다.

#### TC-NLP-010 반말/평서체 위주의 강의 발화
- **대상 항목**: `speech_style_consistency`
- **목적**: 반말 또는 낮춤체 텍스트에서 `informal` 비율이 증가하는지 검증한다.
- **입력 데이터**:
  - “해볼게”, “보면 돼”, “이렇게 가는 거야” 형태의 텍스트
- **기대 결과**:
  - `informal` 비율이 유의미하게 상승한다.
- **실패 조건**:
  - 전반 화법이 명확한데 비율이 중립적으로 산출된다.

#### TC-NLP-011 존댓말과 반말이 혼용된 발화
- **대상 항목**: `speech_style_consistency`
- **목적**: 화법 혼용 상황에서 비율이 실제 혼용 정도를 반영하는지 검증한다.
- **입력 데이터**:
  - 강의 중간에 화법 전환이 여러 차례 발생하는 텍스트
- **기대 결과**:
  - `formal`, `informal`이 모두 0이 아닌 값으로 산출된다.
  - 두 비율의 상대적 크기가 실제 체감 혼용 정도와 크게 어긋나지 않는다.
- **실패 조건**:
  - 한쪽 비율이 일괄 0으로 고정된다.
  - 미세 전환을 과도하게 확대 또는 축소한다.

#### TC-NLP-012 인용문/채팅 낭독이 포함된 발화
- **대상 항목**: `speech_style_consistency`
- **목적**: 강사의 기본 화법과 외부 인용 발화가 구분 없이 혼합 집계되지 않는지 검증한다.
- **입력 데이터**:
  - “질문에 이렇게 써주셨어요” 이후 채팅을 그대로 읽는 구간 포함
- **기대 결과**:
  - 정의된 처리 규칙에 따라 비율 왜곡이 최소화된다.
- **실패 조건**:
  - 채팅 낭독 몇 줄 때문에 전체 강의 화법 비율이 급변한다.

# 5.4 발화 속도 적절성

- **항목 ID**: `speech_rate_appropriateness`
- **관련 JSON 경로**: `concept_clarity_metrics.speech_rate_wpm`
- **관련 rubric 항목**: 타임스탬프 기준 분당 발화량이 수강생이 따라가기 적절한 수준인가.

### 검증 포인트

1. `speech_rate_wpm` 계산식이 고정되어 있는가.
2. 단어 수 정의가 한국어 전사 기준에 맞게 정리되어 있는가.
3. 타임스탬프 누락 구간 처리 기준이 존재하는가.
4. 긴 침묵, 코드 입력, 판서 시간이 발화 속도를 왜곡하지 않는가.

### 대표 테스트 케이스

#### TC-NLP-013 정상 발화 속도의 강의 구간
- **대상 항목**: `speech_rate_appropriateness`
- **목적**: 일반적인 강의 속도에서 적정 범위의 WPM이 산출되는지 검증한다.
- **입력 데이터**:
  - 발화량과 지속 시간이 균형적인 구간
- **기대 결과**:
  - `speech_rate_wpm`이 사전 정의한 일반 범위 내에 위치한다.
- **실패 조건**:
  - 정상 구간인데 비정상적으로 높거나 낮은 값이 산출된다.

#### TC-NLP-014 매우 빠른 설명 구간
- **대상 항목**: `speech_rate_appropriateness`
- **목적**: 빠른 발화에서 WPM이 상승하는지 검증한다.
- **입력 데이터**:
  - 짧은 시간 안에 고밀도 설명이 이어지는 구간
- **기대 결과**:
  - `speech_rate_wpm`이 TC-NLP-013보다 높다.
- **실패 조건**:
  - 실제 체감보다 낮게 계산된다.

#### TC-NLP-015 긴 정적 구간 또는 코드 입력 구간 포함
- **대상 항목**: `speech_rate_appropriateness`
- **목적**: 무발화 시간이 있는 경우 속도 계산 왜곡 여부를 검증한다.
- **입력 데이터**:
  - 강사가 코드를 입력하거나 잠시 멈추는 시간이 포함된 타임스탬프 데이터
- **전처리/가정**:
  - 무발화 구간을 전체 시간에 포함할지 제외할지 정책을 사전 고정한다.
- **기대 결과**:
  - 정책에 맞는 일관된 속도값이 산출된다.
- **실패 조건**:
  - 동일 데이터라도 무발화 처리에 따라 실행 결과가 불안정하다.

#### TC-NLP-016 타임스탬프 누락/역전 데이터
- **대상 항목**: `speech_rate_appropriateness`
- **목적**: 비정상 시간 정보 입력 시 안정적으로 예외 처리하는지 검증한다.
- **입력 데이터**:
  - 종료 시각이 시작 시각보다 빠르거나 일부 구간 시간이 없는 데이터
- **기대 결과**:
  - 오류 로그 또는 정의된 예외 처리 방식으로 대응한다.
- **실패 조건**:
  - 음수 시간, 무한대, NaN WPM 출력

# 5.5 이해 확인 질문

- **항목 ID**: `understanding_question`
- **관련 JSON 경로**: `interaction_metrics.understanding_question_count`
- **관련 rubric 항목**: 수강생의 이해 여부를 확인하는 질문을 적절히 하는가.

### 검증 포인트

1. 이해 확인 질문 패턴이 정의되어 있는가.
2. 단순 수사적 질문과 실제 이해 점검 질문을 구분하는가.
3. `understanding_question_count`가 수작업 정답셋과 대체로 일치하는가.
4. 동일 질문의 중복 집계 또는 누락이 없는가.

### 대표 테스트 케이스

#### TC-NLP-017 이해 확인 질문이 거의 없는 설명 중심 발화
- **대상 항목**: `understanding_question`
- **목적**: 질문이 없는 강의 구간에서 count가 낮게 유지되는지 검증한다.
- **입력 데이터**:
  - 설명 중심의 일방향 발화 텍스트
- **기대 결과**:
  - `understanding_question_count`가 0 또는 매우 낮은 값이다.
- **실패 조건**:
  - 평서문을 질문으로 과다 검출한다.

#### TC-NLP-018 이해 확인 질문이 반복적으로 등장하는 발화
- **대상 항목**: `understanding_question`
- **목적**: “이해되셨죠?”, “여기까지 괜찮으세요?” 같은 질문이 정확히 집계되는지 검증한다.
- **입력 데이터**:
  - 이해 점검 질문이 여러 번 포함된 텍스트
- **기대 결과**:
  - 질문 수가 수작업 집계값과 일치하거나 허용 오차 내에 있다.
- **실패 조건**:
  - 질문 count 누락 또는 중복 집계

#### TC-NLP-019 수사적 질문과 실제 이해 확인 질문 혼재
- **대상 항목**: `understanding_question`
- **목적**: “왜 그럴까요?” 같은 설명용 질문과 “이해되셨나요?” 같은 이해 점검 질문을 구분하는지 검증한다.
- **입력 데이터**:
  - 설명 전개용 질문과 수강생 점검 질문이 섞인 텍스트
- **기대 결과**:
  - 실제 이해 확인 질문만 count에 반영되거나, 정의된 규칙대로 분리 집계된다.
- **실패 조건**:
  - 모든 물음표 문장을 동일하게 집계한다.

#### TC-NLP-020 질문형 종결 어미가 없는 이해 확인 발화
- **대상 항목**: `understanding_question`
- **목적**: “여기까지 따라오시면 됩니다”, “한번 체크해보세요”처럼 물음표 없이 이해 확인 기능을 수행하는 표현 처리 기준을 검증한다.
- **입력 데이터**:
  - 명령형/권유형 이해 점검 발화 포함 텍스트
- **기대 결과**:
  - 사전 정의한 규칙에 따라 포함 여부가 일관되게 결정된다.
- **실패 조건**:
  - 유사 표현이 실행마다 다르게 집계된다.
